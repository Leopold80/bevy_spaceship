当前 MuJoCo 实验仍是零重力、固定质量的 Apollo 外形单刚体，但已同时提供理想
body-frame 6D wrench plant 和独立的 Apollo 11 RCS + DPS 推进器 plant。后者已包含 16
路喷口站位、点力、最小脉冲和瞬态近似；它仍不是含月球环境与变质量的完整登月动力学。

长期直接输出每一个喷口的开关指令。

## 核心原则

Apollo 风格飞船上有很多 RCS 喷口，但姿态控制器不应该直接把所有喷口都当成独立控制输入。控制律更合理的输出是期望力矩，或者更一般地说，是期望六维广义力：

```text
attitude error -> desired torque -> thruster allocation -> thruster firing commands
```

姿态控制器主要关心：

```text
tau_c = [tau_x, tau_y, tau_z]^T
```

而不是直接关心：

```text
u = [u_1, u_2, ..., u_N]^T
```

其中 `N` 是 RCS 喷口数量。喷口数量多，主要是为了提供正负方向力矩、平移能力、力矩偶构造、冗余容错，以及适应质心变化和构型约束；它不意味着姿态控制问题本身应该被建模成一个高维连续控制输入问题。

## 喷口效能矩阵

第 `i` 个喷口可以用两个 body-frame 向量描述：

```text
r_i: 喷口相对质心的位置
f_i: 喷口点火时产生的力方向和最大推力
```

它对刚体产生的力矩为：

```text
tau_i = r_i x f_i
```

如果只考虑姿态控制，可以构造力矩效能矩阵：

```text
B_tau = [tau_1, tau_2, ..., tau_N]
```

于是有：

```text
tau = B_tau u
```

如果同时考虑平移和姿态，则应构造六维广义力效能矩阵：

```text
w_i = [f_i; r_i x f_i]
B_w = [w_1, w_2, ..., w_N]
w = B_w u
```

其中：

```text
w = [F_x, F_y, F_z, tau_x, tau_y, tau_z]^T
```

这既对应理想 MuJoCo plant 的 body-frame 6D 输入，也可由当前 RCS 规格直接计算。推进器
plant 已按具体喷口施力；尚未完成的是把上层 `w_c` 或 `tau_c` 自动分配成 16 路命令。

## 开关型执行器与脉冲调制

RCS 喷口通常不是连续可调推力，而是开关型执行器。因此单个喷口的控制量更接近：

```text
u_i in {0, 1}
```

在离散控制周期内，也可以用占空比或点火时长近似：

```text
u_i in [0, 1]
```

因此，RCS 控制分配不是普通的连续控制输入映射，而是包含以下约束的执行器分配问题：

- 喷口只能开/关，或者只能在一个控制周期内用有限脉宽近似连续推力。
- 喷口有最小点火脉宽，不能无限短地抖动。
- 喷口有最大占空比和热约束。
- 某些喷口可能因为构型、故障或喷流干涉而不可用。
- 同一组喷口可能需要避免不必要的频繁切换。

所以实际实现中需要死区、滞环、最小脉宽、脉冲调制和喷口选择逻辑。

## 姿态力矩与平动解耦

对于姿态控制，理想情况是产生期望力矩，同时尽量避免引入不必要的净平动力：

```text
sum f_i ~= 0
sum r_i x f_i = tau_c
```

例如要产生一个纯 yaw 力矩，最好选择一对或几对喷口，使它们的平动力相互抵消，而力矩方向相加。这样飞船主要转动，不会因为姿态修正而产生明显平移漂移。

这也是多喷口布局的工程意义之一：喷口冗余不是为了让上层控制器直接输出高维喷口指令，而是为了让分配层在不同控制目标和约束之间进行选择。

## 建议的软件分层

后续可以把控制系统分成三层：

```text
Guidance / Reference
    -> Attitude Controller
    -> Thruster Allocator
    -> MuJoCo RCS Actuator Model
```

在本项目中，可以先保持现有双层姿态控制器：

```text
outer loop: q_e -> omega_c
inner loop: omega_c - omega_body -> tau_c
```

然后在 `tau_c` 后面增加 RCS 分配层：

```text
tau_c -> allocator -> thruster commands -> MuJoCo forces
```

这样姿态控制器仍然是低维、清晰、可验证的；复杂的喷口数量、开关约束和冗余选择由执行器层处理。

## 建议的数据结构

可以先抽象一个喷口规格：

```text
ThrusterSpec {
    position_body,
    direction_body,
    max_force,
    min_on_time,
    group_id,
    enabled,
}
```

随后由喷口规格自动生成效能矩阵：

```text
build_wrench_matrix(thrusters, center_of_mass) -> B_w
```

控制分配器可以提供类似接口：

```text
allocate_torque(tau_c, state, dt) -> Vec<ThrusterCommand>
allocate_wrench(w_c, state, dt) -> Vec<ThrusterCommand>
```

其中 `ThrusterCommand` 可以包含：

```text
thruster_id
on_time
duty_ratio
```

早期实现可以只支持姿态力矩分配，后续再扩展到六维平移/姿态统一分配。

## 渐进实现路线

1. **继续保留当前 body-frame 6D 输入**  
   这是验证刚体动力学和控制律的基线，不要过早删除。

2. **定义 Apollo 风格 RCS 喷口规格**  
   先在 body frame 中定义喷口位置、方向、最大推力和所属 quad/group。

3. **生成效能矩阵**  
   根据喷口规格和质心位置生成 `B_tau` 或 `B_w`。

4. **实现连续近似分配器**  
   先允许 `u_i in [0, 1]`，用最小二乘或伪逆得到一个可解释的初始分配结果。

5. **加入离散喷口选择逻辑**  
   从连续分配结果过渡到开关点火，加入死区、滞环、最小脉宽和占空比限制。

6. **加入力矩偶优先规则**  
   对 roll/pitch/yaw 纯姿态控制，优先选择净平动力较小的喷口组合。

7. **接入 MuJoCo 执行器模型**  
   逐步从“直接施加 6D 广义力”切换到“在喷口位置施加喷口力”。可考虑用 site/actuator 或等效的外力施加方式表达每个喷口。

8. **增加退化与故障测试**  
   测试单个喷口失效、某个 quad 失效、质心偏移、最小脉宽过大等情况下的控制效果。

## 最小验证项

在引入 RCS 分配层后，应至少检查：

- 给定纯 roll/pitch/yaw 力矩指令时，分配结果的净平动力是否足够小。
- `B_tau u` 或 `B_w u` 是否接近期望指令。
- 最小脉宽和死区不会造成高频抖振。
- 控制器输出维度仍然保持为 `tau_c` 或 `w_c`，而不是扩展成直接控制所有喷口。
- 禁用部分喷口后，分配器能够给出降级解，或者明确报告当前指令不可实现。

## 一句话总结

RCS 喷口数量很多，但上层姿态控制器不应该直接控制每个喷口。合理结构是：控制器输出低维期望力矩或六维广义力，控制分配器再根据喷口布局、开关约束和冗余规则，把它转换为具体喷口的点火命令。
